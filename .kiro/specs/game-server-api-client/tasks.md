# Implementation Plan

## Branches

**Base**: `feature/game-server-api-client`

| Branch | Tasks | Goal |
|--------|-------|------|
| `feature/game-server-api-client-client` | 1-2 | APIクライアントがHTTPリクエストを送受信でき、構造化された結果型を返す状態 |
| `feature/game-server-api-client-error-handling` | 3-4 | エラーハンドリングサービスによるリトライ制御・ダイアログ連携が動作し、ユニットテストが通る状態 |

## Tasks

### Branch: `feature/game-server-api-client-client`

- [x] 1. API通信の結果・エラー情報を表す共通データモデルと接続設定を定義する
  - ネットワークエラー、タイムアウト、認証エラー、クライアントエラー、サーバエラー、パースエラーの6種類のエラー種別を定義する
  - エラー種別、ステータスコード、メッセージ、レスポンスボディ、リトライ可否を保持するエラー情報モデルを作成する
  - リトライ可否の判定ロジックを実装する（ネットワークエラー・タイムアウト・サーバエラーはリトライ可能、それ以外はリトライ不可）
  - 成功またはエラーを表すジェネリック型の通信結果モデルを作成し、ファクトリメソッドで生成可能にする
  - ベースURLをコンパイルディレクティブで環境切り替え可能にし、タイムアウト秒数と合わせて定数として定義する
  - _Requirements: 3.1, 3.2, 3.3, 3.4, 5.1, 5.2_

- [x] 2. ゲームサーバとのHTTP通信・認証・エラー分類・ログ出力を行うAPIクライアントを実装する
  - [x] 2.1 HTTPリクエスト送受信をUniTask非同期パターンで実装する
    - GET・POST・PUT・DELETEの各HTTPメソッドに対応する非同期メソッドを提供する
    - リクエストボディのJSONシリアライズとレスポンスボディのJSONデシリアライズを行う
    - Content-Typeヘッダーを自動付与する
    - すべてのメソッドでCancellationTokenによるキャンセルをサポートする
    - タイムアウト制御を実装し、タイムアウト時にリクエストを中断する
  - [x] 2.2 Bearer token管理による認証機能を実装する
    - トークンの保持と認証付きリクエストへのAuthorizationヘッダー自動付与を行う
    - トークンの設定・クリアメソッドを提供する
    - トークン未設定時は公開APIとしてAuthorizationヘッダーなしでリクエストを送信する
  - [x] 2.3 エラー分類とAPI通信ログ出力を実装する
    - HTTP通信例外をエラー種別に分類し、構造化された結果型として返却する
    - タイムアウト例外を検出してタイムアウトエラーとして分類する
    - JSONデシリアライズ失敗時をパースエラーとして分類する
    - すべてのエラーをクラスコンテキスト付きでDebug.LogErrorに出力する
    - 開発ビルドでのみリクエスト（メソッド、エンドポイント、ボディ）とレスポンス（ステータスコード、ボディ、レスポンスタイム）をDebug.Logに出力する
    - レスポンスタイムの計測をログ出力ディレクティブ内に配置する
  - [x] 2.4 APIクライアントをRootScope DIコンテナにSingletonとして登録する
    - インターフェースを通じた抽象化を提供し、依存方向ルールに従って登録する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 2.1, 2.2, 2.3, 2.4, 3.5, 3.6, 6.1, 6.2, 6.3, 6.4, 7.1, 7.2, 7.5_

### Branch: `feature/game-server-api-client-error-handling`

- [ ] 3. API通信エラーに対するダイアログ表示・リトライ制御・タイトル遷移を一元管理するエラーハンドリングサービスを実装する
  - [ ] 3.1 API呼び出しデリゲートの実行と成功時の結果返却を実装する
    - CancellationToken付きのAPI呼び出しデリゲートを受け取り実行する
    - 成功時は結果をそのまま呼び出し元に返却する
  - [ ] 3.2 リトライ可能エラー時のユーザー選択ダイアログとリトライループを実装する
    - リトライ可能エラー発生時に「リトライ」と「タイトルに戻る」の選択肢を持つ確認ダイアログを表示する
    - ユーザーがリトライを選択した場合はデリゲートを再実行する（成功するまでループ可能）
    - ユーザーがタイトルを選択した場合はタイトル画面に遷移する
  - [ ] 3.3 リトライ不可エラー時のエラー通知とタイトル遷移を実装する
    - リトライ不可エラー発生時にエラーメッセージダイアログを表示する
    - ダイアログ確認後にタイトル画面に遷移する
    - タイトル遷移後にOperationCanceledExceptionをスローして呼び出し元のスタックを巻き戻す
  - [ ] 3.4 エラーハンドリングサービスをRootScope DIコンテナにSingletonとして登録する
    - インターフェースを通じた抽象化を提供し、依存方向ルールに従って登録する
  - _Requirements: 4.1, 4.2, 4.3, 4.4, 4.5, 4.6, 4.7, 7.3, 7.4, 7.5_

- [ ] 4. データモデルとエラーハンドリングロジックのユニットテストを実装する
  - [ ] 4.1 (P) 通信結果型とエラー情報モデルのテストを実装する
    - Success/Failureの構築とプロパティアクセスを検証する
    - 全エラー種別に対するリトライ可否判定の正確性を検証する（リトライ可能: ネットワークエラー・タイムアウト・サーバエラー、リトライ不可: 認証エラー・クライアントエラー・パースエラー）
  - [ ] 4.2 (P) エラーハンドリングサービスの振る舞いをモックを用いてテストする
    - 成功時に結果がそのまま返却されることを検証する
    - リトライ可能エラー時に確認ダイアログが表示され、リトライ選択でデリゲートが再実行されることを検証する
    - リトライ可能エラー時にタイトル選択でタイトル画面に遷移することを検証する
    - リトライ不可エラー時にメッセージダイアログ表示後タイトル画面に遷移することを検証する
    - IDialogServiceとSceneLoaderをモックして依存を分離する
  - _Requirements: 3.1, 3.2, 3.4, 4.1, 4.2, 4.3, 4.4, 4.5, 4.6_
