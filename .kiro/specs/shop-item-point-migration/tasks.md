# Implementation Plan

## Branches

**Base**: `feature/shop-item-point-migration`

全タスクを base ブランチで実装する。`ShopState` の API 撤去、`GachaData` の型変更、`ShopService` / `ShopView` の依存差し替えは相互にコンパイル上密結合するため、単一 PR として main へ統合する。

## Tasks

### Branch: `feature/shop-item-point-migration`

- [x] 1. ガチャ報酬データを uint 家具 ID 体系へ移行
  - [x] 1.1 ガチャデータの報酬家具 ID を `uint` の読み取り専用リストに置き換える
    - 既存の文字列 ID (`"chair_01"` など) は廃止し、`IUserItemInventoryService.AddFurniture` が受理する `uint` 型と整合させる
    - データ契約としての列挙を `IReadOnlyList<uint>` に統一する
  - [x] 1.2 ショップ起動時のモックデータ投入を uint リテラルに更新する
    - 既存のモック商品・ガチャ投入ロジック内のリテラルを uint に差し替える
    - マスター未整備 ID を含み得る前提とし、後続タスクの縮退処理で吸収する
  - _Requirements: 7.1, 7.2_

- [x] 2. ShopState を UI 固有状態のみへ縮退
  - [x] 2.1 毛糸残高・消費・加算の API と残高変更イベントを撤去する
    - 毛糸残高プロパティ、消費メソッド、加算メソッド、残高変更イベントを `ShopState` から削除する
    - 削除によって発生するコンパイルエラーを頼りに、後続タスクで呼び出し元を洗い出せる状態にする
  - [x] 2.2 タブ状態と商品モックリストのみを保持する構成に整理する
    - 残す API はショップシーン固有の状態 (タブ・商品/ガチャリスト) に限定する
    - 毛糸残高や家具所持数のキャッシュをフィールドとして保持しないことを担保する
  - _Requirements: 1.5, 2.6, 3.5, 4.5, 4.6, 8.1, 8.2, 8.5_

- [x] 3. ShopService に RootScope サービスを注入
  - [x] 3.1 コンストラクタへ毛糸サービス・家具インベントリサービス・マスターデータ参照を追加する
    - VContainer のコンストラクタ注入を利用し、IL2CPP stripping 対策として注入属性を付与する
    - 既存の状態参照・ダイアログ参照・シーン遷移参照と並べ、依存一覧を設計書と整合させる
  - [x] 3.2 ShopScope が RootScope 契約を再登録しないことを検証する
    - 親スコープ継承により毛糸サービス・家具インベントリサービスが解決できることを前提化する
    - 依存方向が View → Service → State のまま維持され、State から Service/View への参照が発生しないことを確認する
  - _Requirements: 9.1, 9.3, 9.4_

- [ ] 4. 残高参照とセル購入可否判定を RootScope サービスへ接続
  - [ ] 4.1 商品セルとガチャセルの購入可否判定を毛糸サービス残高に基づく計算へ差し替える
    - 毛糸通貨商品の購入可否は「毛糸残高 ≥ 商品価格」の判定に統一する
    - ガチャの単発/10 連ボタンの購入可否判定も毛糸残高を直接参照する経路に切り替える
  - [ ] 4.2 リアルマネー商品は残高非依存で常に購入可のまま扱う既存挙動を維持する
    - 毛糸残高に依存せず購入可能扱いとする分岐を明示し、既存体験を退行させない
  - _Requirements: 1.1, 1.3, 5.1, 5.2, 5.3, 5.4_

- [ ] 5. 毛糸通貨商品の購入フローを RootScope サービスへ移行
  - [ ] 5.1 購入確認ダイアログを開く前に残高を検証し、不足時はメッセージダイアログのみを表示する
    - 確認ダイアログ表示前に毛糸残高と価格を比較し、不足時は「毛糸が足りません」相当のメッセージを提示して確認ダイアログを開かない
  - [ ] 5.2 毛糸消費結果コードを判定し、結果ごとに通知・ログ・中止を出し分ける
    - Insufficient はユーザー通知 (メッセージダイアログ) を行い、後続購入処理を中止する
    - InvalidArgument はクラスコンテキスト付きのエラーログのみ出力し、ユーザー通知は行わず中止する
  - [ ] 5.3 毛糸パック成功時の残高加算と失敗時の縮退を実装する
    - 成功時に `YarnAmount` 分を毛糸サービスへ加算する
    - Overflow は購入完了メッセージ内に加算失敗を付記しつつ購入自体は成立扱いとする
    - InvalidArgument (未設定値など) はログ出力のみで加算をスキップする
  - [ ] 5.4 購入完了メッセージと確認ダイアログ No 時の無操作を既存挙動で維持する
    - 従来と同等の「〜を購入しました！」系メッセージを表示する
    - 確認ダイアログで No を選択した場合は残高・インベントリ・状態いずれも操作しない
  - _Requirements: 2.1, 2.3, 2.4, 2.5, 3.1, 3.2, 3.3, 3.4, 6.1, 6.2, 6.5, 10.2, 10.3_

- [ ] 6. ガチャ購入フローと家具加算を連携
  - [ ] 6.1 ガチャ確認ダイアログ表示前に残高を検証し、不足時はメッセージダイアログで中断する
    - 単発/10 連それぞれの価格と毛糸残高を比較し、不足時は確認ダイアログを開かない
  - [ ] 6.2 ガチャ成立時の毛糸消費結果コードを判定し、通知・ログ・中止を出し分ける
    - Insufficient はユーザー通知、InvalidArgument はクラスコンテキスト付きのエラーログで中止する
  - [ ] 6.3 抽選結果の家具 ID 群を家具インベントリ API で 1 件ずつ加算する
    - 抽選回数 (単発=1 / 10 連=10) ぶんの家具 ID を生成し、ID ごとに所持数を +1 する
  - [ ] 6.4 家具加算の UnknownId / InvalidArgument を部分失敗として扱う
    - 該当 ID のみスキップし、残りの抽選結果処理を継続する
    - いずれの場合もクラスコンテキスト付きのエラーログを残す
  - [ ] 6.5 結果メッセージの家具名をマスターデータから解決し、未登録 ID は ID 表記で縮退する
    - マスターデータから家具名を引当てて表示する
    - マスター未登録 ID の場合は uint の ID をそのまま表示するフォールバックを備える
  - _Requirements: 2.2, 6.3, 6.4, 7.1, 7.2, 7.3, 7.4, 7.5_

- [ ] 7. ShopView の残高表示と購読経路を切替
  - [ ] 7.1 Construct に毛糸サービスを追加し注入属性を付与する
    - VContainer の Construct 注入経由で毛糸サービスを受け取れるようにする
  - [ ] 7.2 Start で毛糸サービスの現在残高を初期表示し、残高変更通知を購読する
    - 初期化時点の残高を画面表示に反映し、以降はイベント駆動で更新を受ける
    - 解除漏れを防ぐため、ハンドラはメソッド参照で登録する
  - [ ] 7.3 残高変更ハンドラで残高表示と全セル購入可否表示を再計算する
    - 残高テキスト更新と全商品セル・ガチャセルの購入可否再計算を単一経路で行う
    - 購入完了メッセージ側で手動の表示更新を行わない方針を徹底する
  - [ ] 7.4 OnDestroy で残高変更通知の購読を解除し、Singleton 側へのリークを防ぐ
    - ショップシーンの破棄または View 破棄時に購読解除を確実に実行する
  - [ ] 7.5 ShopState 由来の残高購読と参照を撤去する
    - 残高関連のフィールド/イベント参照を ShopView から完全に取り除く
  - _Requirements: 1.2, 1.4, 4.1, 4.2, 4.3, 4.4, 8.4, 9.2, 10.1, 10.4_

- [ ] 8. 起動時初期化とシーン再訪問時の挙動を検証
  - [ ] 8.1 ShopStarter 経由の Initialize が毛糸サービス前提でモック商品リストを投入できることを確認する
    - SceneScope.Awake によりマスターデータ取込が完了した状態で Initialize が呼ばれる前提を保つ
    - 初期化処理が残高参照を伴わずに完了することをエディタ上で確認する
  - [ ] 8.2 シーン離脱後に再訪問した際、毛糸サービスの残高と購読が正しく反映されることを確認する
    - シーン再入場で前回の毛糸消費結果が表示に反映されることを手動検証する
    - 二重購読や購読解除漏れが発生しないことを併せて確認する
  - _Requirements: 9.5, 10.5_

    - Acceptance Criteria (Requirement 3.1) を単体テストで固定化する

    - Acceptance Criteria (Requirement 7.3) を単体テストで固定化する
  - [ ]* 9.4 リアルマネー商品が残高非依存で購入可になるケースを検証する
    - Acceptance Criteria (Requirement 5.4) を単体テストで固定化する
